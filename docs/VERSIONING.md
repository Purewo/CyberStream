# 版本管理规范

本文档用于统一赛博影视项目的版本号使用方式、升级规则与同步要求。

## 1. 单一版本源

项目统一版本号来源为：

- `backend/config.py` 中的 `APP_VERSION`

当前版本：

- `1.17.0`

状态：

- 当前 `main` 主干最新版
- 暂不维护长期开发分支，小步提交直接进入 `main`
- `1.16.0` 保留为历史稳定标签 `v1.16.0`

除特殊说明外，以下位置都应以该字段为准：

- 健康检查接口 `/`
- README 中的当前版本说明
- 接手文档中的当前版本说明
- 运行与验收说明中的预期版本

---

## 2. 版本号格式

采用三段式语义版本风格：

```text
主版本.次版本.修订版本
MAJOR.MINOR.PATCH
```

例如：

- `1.11.0`
- `1.11.1`
- `1.12.0`
- `2.0.0`

---

## 3. 升级规则

### 3.1 PATCH（修订版本）
适用于：

- Bug 修复
- 文档修正但希望保留发布痕迹
- 小范围兼容性修复
- 不改变接口含义的小调整

示例：
- `1.11.0 -> 1.11.1`

### 3.2 MINOR（次版本）
适用于：

- 新增接口
- 新增筛选项或能力
- 扩展存储协议支持
- 新增可感知功能，但尽量保持兼容

示例：
- `1.11.0 -> 1.12.0`

### 3.3 MAJOR（主版本）
适用于：

- 破坏性接口变更
- 数据模型有明显不兼容调整
- 前端需要明显改造才能适配
- 配置结构发生破坏性变化

示例：
- `1.11.0 -> 2.0.0`

---

## 4. 每次升级必须同步的地方

至少检查以下内容：

1. `backend/config.py` 中的 `APP_VERSION`
2. 健康检查 `/` 返回值是否正确
3. `README.md` 当前版本说明
4. `docs/PROJECT_HANDOVER.md` 当前版本说明
5. `docs/RUNBOOK.md` 中的验收预期版本
6. 如本次改动较大，应补充 `docs/TEST_CHECKLIST.md` 中的已验证项

---

## 5. 推荐发布流程

每次正式完成一轮功能或修复后，建议按下面顺序执行：

1. 修改代码
2. 本地启动服务
3. 验证健康检查与相关接口
4. 按改动类型决定升 PATCH / MINOR / MAJOR
5. 更新 `APP_VERSION`
6. 同步 README / 接手文档 / Runbook
7. 按 `docs/TEST_CHECKLIST.md` 做本地与公网验收

---

## 6. 当前建议

在项目仍处于接手整理阶段时：

- 文档补齐、兼容性修复、低风险工程性修补：优先升 `PATCH`
- 新增功能：升 `MINOR`
- 暂不建议轻易升 `MAJOR`

---

## 7. 重要约定

- 不要在多个文件中手写不同版本号
- 版本号必须以 `APP_VERSION` 为唯一真值
- 代码注释中的历史 `v1.x.x` 记录可暂时保留，但不应再作为实际运行版本依据
